## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                        【Arduino UNO R3】                       【出力】
ボタン ──┬──────────────→┐
         │               │
         └─[リセット信号]─┘
（リセット/モード切替）      │   タイマー管理・状態遷移   ├──→ LED（リマインド表示、電源表示）
                             └───────────────────────┘   └──→ アクティブブザー（警告）
```

補足説明（図で表現しきれない場合）：
- ボタン入力でリマインダーのリセットやモード切替を行う。
- タイマーで一定時間ごとにLEDを点滅させてリマインド。
- LED点滅後30秒間反応がなければブザーで警告。
- すべての制御はArduino UNO R3上で行う。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（リマインド時間経過）──→ [リマインド中] ──（30秒無反応）──→ [警告中]
   ↑         └─（ボタン押下）────────────┘                              │
   |               ↓                                                   |
   └────────（ボタン押下）──────────────┘                               │
                ↑──────────────────────（ボタン押下）───────────────────┘
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中 | タイマーでリマインド時間を計測。ボタン押下でタイマーをリセットし、黄色LEDを一定時間点滅させる | 緑LED点灯、黄色LED消灯（フィードバック時のみ黄色LED点滅） |
| リマインド中 | 黄色LEDで水分補給を促す | 緑LED点灯、黄色LED点灯 |
| 警告中 | 30秒間反応がなかった場合警告。ボタン押下で警告解除・待機に戻る | 緑LED点灯、黄色LED点灯のまま、開始時にブザー2回鳴動しその後は一定間隔で再鳴動 |

※ 電源ON中は緑LEDを常時点灯し、電源OFF時は緑LED・黄色LEDともに消灯する。

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| LED表示（電源ON中） | 緑LEDは常時点灯 | **millis** | 電源ON状態の視覚表示を常に維持しながら他処理を並行するため |
| リマインド開始判定 | 30分（1800000ms） | **millis** | 長時間待機中も他処理を止めないため |
| 警告開始判定（無反応） | 30秒（30000ms） | **millis** | リマインド中の入力監視を継続するため |
| 警告中の再鳴動 | 30秒（30000ms）ごと | **millis** | 開始時2回鳴動＋以降も毎回2回鳴動をnon-blockingで再通知するため |
| ボタンデバウンス判定 | 50ms | **millis** | チャタリングを除去しつつ応答性を保つため |
| 待機中ボタン押下の黄色LED点滅 | 1秒（1000ms） | **millis** | タイマーリセット時の視覚フィードバックを返すため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合：D2（ボタン）、D9（緑LED）、D10（黄色LED）、D12（アクティブブザー）で競合なし。3-1で明記済み。
- ライブラリ競合：標準関数のみで特別なライブラリ不要。競合リスクなし。
- 状態遷移：待機→リマインド→警告→待機のループが明確で、全状態からリセット可能。抜けなし。
- delay()の問題点：全てのタイミング制御はmillis()で設計されており、delay()による処理停止リスクなし。

**対応した内容：**
- ピン割り当て・競合チェックを3-1で明記
- 状態遷移図・表で全状態の遷移・リセットルートを明記
- タイミング設計でdelay()を使わずmillis()を使用

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
