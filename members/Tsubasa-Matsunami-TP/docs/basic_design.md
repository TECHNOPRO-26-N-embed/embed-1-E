## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ (グー)─→─┐                             ┌-→LED
タクトスイッチ (チョキ)→┤　じゃんけん処理・             ├─→ブザー
タクトスイッチ (パー)─→─┘　ランダム処理・状態管理       ┘

```

補足説明（図で表現しきれない場合）：
３つのタクトスイッチを使用して、それぞれのグー・チョキ・パーを入力する
Arduino側でランダムに手を生成し、プレイヤー側との勝利判定を行う
判定結果に応じてLEDやブザーで結果を出力する

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（ 初期化完了 ）
[入力待機] ──（ボタン入力）──→ [勝敗判定] ──（判定完了）──→ [結果表示]
  　↑        │                    │              │
    │        └──(あいこの場合)────┘               │
    │                                            │
    └─────────(約３秒経過)────────────────────────┘

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 初期化中 | ピン設定や変数初期化など、ゲーム開始前の準備を行う | LED消灯、短い開始音を鳴らす |
| 入力待機 | プレイヤーのグー・チョキ・パー入力を待機する | LEDを点灯して待機状態を表示 |
| 勝敗判定 | Arduino側でランダムに手を決定しプレイヤー側との勝敗を判定する | LED状態は維持 |
| 結果表示 | 勝ち・負け・あいこの結果を表示する | 勝敗に応じてLED点灯やブザー音を変更する |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-2. 機能一覧・関数一覧（役割の定義）

> 具体的な実装は詳細設計書（detailed_design.md）で行います。

> [!TIP]
> **単一責任の原則（1関数 = 1つの仕事）**
> 「機能名」欄に requirements.md の機能名（必須・追加）を書き、それを実現する関数名を対応させてください。
> 入出力の読み取り（readButton 等）や制御（updateOutput 等）は複数機能から使われる共通処理として「共通」と書いてOKです。

| 機能ID | 機能名 | 関数名 | 担う「1つの仕事」 | 主な引数 | 戻り値 | 呼び出す場所 |
|:--|:--|:--|:--|:--|:--|:--|
|  | 初期化 | `setup()` | ピンモード設定や初期状態の設定を行う | なし | なし | 起動時一回 |
|  | (共通) 待機・制御 | `loop()` | 状態に応じて各関数を呼び出す | なし | なし | 常時 |

|  | (共通) 勝敗判定 | `judgeResult()` | プレイヤーとCPUの手から勝敗を判定する | int playHand, int cpuHand | int | loop()内 |

> loop() の中で全部書こうとしていませんか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| ボタン入力監視 | 常時 | millis | 入力を常に監視し、押し逃しを防ぐため |
| チャタリング対策 | 50ms | millis | 一定時間内の連続入力を制御するため |
| 勝敗結果表示 | 約2秒 | millis | 結果表示中でも状態管理を継続するため |
| LED点灯制御 | 状態ごと | millis | 他処理と並行して制御するため |
| ブザー出力 | 状態ごと | millis | LEDや入力処理と同時に動作させるため |
| 起動時の初期待機 | 約1秒 | delay | 起動時のみの単純な待機処理であり、他処理へ影響が少ないため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

### 3-1. 部品一覧とピン接続

> ※ デジタルピン: D2〜D13 / アナログピン: A0〜A5
> 特殊用途: D0/D1（シリアル通信・使用禁止）、A4/A5（I2C）、D10〜D13（SPI）

| 部品名 | 型番・種類 | Arduinoピン | 入力 / 出力 | 備考 |
|:--|:--|:--|:--|:--|
| タクトスイッチ(グー) | プッシュボタン | D2 | 入力 | INPUT_PULLUP使用 |
| タクトスイッチ(チョキ) | プッシュボタン | D3 | 入力 | INPUT_PULLUP使用 |
| タクトスイッチ(パー) | プッシュボタン | D4 | 入力 | INPUT_PULLUP使用 |
| LED | 5mmLED | D13 | 出力 | 220Ω抵抗と直列 |
| ブザー | アクティブブザー | D8 | 出力 | 勝敗結果の音出力 |
| 抵抗 | 220Ω | - | - | LED保護 |

> [!CAUTION]
> **ピン競合チェック（必須）**
> - D0/D1 は Serial 通信専用のため使用禁止
> - I2C 機器（LCD等）は A4/A5 を使用
> - 同じピンに複数の部品を繋いでいないか確認

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | ボタンを押したとき | 前回入力から50ms未満の入力を検出 | デバウンス処理で入力を無効化 |
| ボタン同時押し | 複数のボタンを同時に押したとき | 2つ以上の入力を同時検知 | 入力を無効として再入力を待つ |
| 入力なし | 一定時間(約３秒)操作がされなかったとき | タイマーで一定時間経過を検出 | 入力状態へ戻る |
| 不正な勝敗値 | 勝敗判定で想定外の値が入ったとき | 判定値が定義範囲外か確認 | エラー扱いとして初期状態へ戻す |
| ブザー出力失敗 | ブザーが正常に鳴らない場合 | 出力信号送信後も状態変化なし | LEDのみで結果表示を継続する |
